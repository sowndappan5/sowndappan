MODEL FINE-TUNING

Commit Message LoRA Fine-Tuning

Problem

Developers often write vague, informal commit messages under pressure (e.g., "fix login bug"). While large language models can rewrite these into clean, professional summaries, deploying a full 7B or 70B parameter model solely for commit formatting is highly cost-inefficient. There is a need for a highly capable but small model that can run efficiently on edge or consumer hardware.

Input  : fix login bug
Output: Fix authentication failure caused by invalid session token handling

Workflow

The pipeline starts by downloading and cleaning the GitHub Commit Messages dataset from Kaggle, then sampling 5,000 rows for training. The data is tokenized and formatted into instruction templates. Using Parameter-Efficient Fine-Tuning (LoRA) and 4-bit bitsandbytes quantization, the Qwen2.5-1.5B-Instruct model is fine-tuned without modifying all base weights. Finally, the optimized LoRA adapters are loaded into a PyTorch inference script capable of generating professional commit summaries on a single GPU.

Quickstart

1. Install dependencies — requires a Kaggle API key at ~/.kaggle/kaggle.json

pip install -r requirements.txt

2. Prepare the dataset — downloads, filters by message length, and samples 5,000 rows to data/prepared.csv

python data/prepare_dataset.py

3. Train — 2 epochs with 4-bit quantization + LoRA; adapter saved to ./commit_lora

python -m src.train

4. Run inference

from src.inference import load_model_for_inference, generate_commit_message
model, tokenizer = load_model_for_inference()
print(generate_commit_message("add dark mode", model, tokenizer))
# → "Add dark mode support with system preference detection"

Fine-tuning a full 1.5B model requires significant GPU memory and time. LoRA inserts small trainable rank-decomposition matrices into the attention layers, cutting trainable parameters by ~99% while achieving comparable task performance — making the project reproducible on a single consumer GPU (e.g. a T4 on Colab).

Challenges

The main challenge was handling the high memory footprint required for fine-tuning even a 1.5B parameter model on limited hardware. Standard full fine-tuning would cause Out-Of-Memory (OOM) errors on a single T4 GPU. Additionally, carefully constructing the prompt templates was crucial to ensure the model didn't hallucinate context that wasn't in the original sloppy commit message.

Trade-offs

By choosing LoRA over full fine-tuning, I traded a slight potential drop in absolute maximum model capacity for a massive ~99% reduction in trainable parameters, enabling practical training on consumer hardware. Using 4-bit NF4 quantization slightly degraded the model precision but allowed the training script to run within the strict VRAM constraints of the T4 GPU.
